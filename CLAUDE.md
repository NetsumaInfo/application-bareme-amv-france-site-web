# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Site vitrine + tutoriels scrollytelling for the **AMV Notation** desktop app (https://github.com/NetsumaInfo/application-bareme-amv-france) or local S:\projet_app\app bareme amv  . French content. SSR via TanStack Start.

## Commands

```bash
npm run dev      # vite dev on :3000
npm run build    # vite build (TanStack Start, target node-server)
npm run start    # node .output/server/index.mjs (post-build SSR server)
npm run preview  # vite preview
```

No test runner, no linter, no typecheck script. Type errors surface at build time via `vite build` (tsconfig `strict`, `noUnusedLocals`, `noUnusedParameters`). Run `npx tsc --noEmit` for a fast type check.

`src/routeTree.gen.ts` is generated by `@tanstack/router-plugin` (Vite plugin via `tanstackStart`) and gitignored — never edit by hand; regenerated on `dev`/`build`.

## Architecture

Three coupled layers — all rendered together on `src/routes/index.tsx` (single-page landing). Editing one in isolation usually breaks the choreography of the others.

### 1. Smooth-scroll bridge — `src/components/SmoothScroll.tsx`

Mounted once in `src/routes/__root.tsx`. Initializes a global Lenis instance and forwards every `scroll` event to `ScrollTrigger.update`. **Every GSAP ScrollTrigger in the app depends on this bridge** — if you add a new pinned section and scroll feels janky, suspect Lenis isn't running (SSR / hydration order).

`gsap.registerPlugin(ScrollTrigger)` is guarded by a module-level `registered` flag. `TutorialStage` has its own identical guard (`stRegistered`). Both are intentional — components can mount in any order.

### 2. Tutorial choreography — `src/components/tutorial/TutorialStage.tsx`

Generic, reusable pin+scrub section. Driven by an array of `TutorialStep` and a `render(stepIndex, progress)` callback. Per step the stage scrolls `stepVH` (default 80) vh, and a fake `Cursor` moves to `targetSelector` queried **inside the mock host**, using `gsap.quickTo` for the x/y tween.

**Layout is mock-first**: the mock is the center stage (~1180px max) and the step list sits in a discrete ~200px sidebar on the right. The mock is the focus; text is intentionally small. Don't enlarge the sidebar back to a 40/60 split — the design intent is "tutoriel immersif, texte discret."

Key constraints when adding/editing steps:

- `targetSelector` must resolve inside the rendered mock. Mocks expose anchors via `data-action="…"` / `data-field="…"` attributes — keep those stable when refactoring mock components.
- The cursor re-positions on `stepIndex` change with **double `requestAnimationFrame`** so the mock can re-render before measuring `getBoundingClientRect`. If you swap a sync render for an async one, the cursor will land on stale geometry.
- A `ResizeObserver` on the mock host calls `ScrollTrigger.refresh()`. Don't replace the host wrapper in `render` — measurements rely on `mockHostRef`.
- Total scroll height = `steps.length * stepVH` vh. Long step lists make pages tall fast.

Each individual tutorial (`CreateProjectTutorial`, `ImportVideosTutorial`, …) lives in `src/components/tutorial/tutorials.tsx` and is just `<TutorialStage steps={…} render={idx => <AppShell><SomeMock state={stateForStep(idx)} /></AppShell>} />`. The pattern: derive per-step state from `idx` (often an array indexed by `idx`), pass to the matching mock.

### 3. App mocks — `src/components/app-mock/*`

Pixel-faithful facsimiles of the desktop app's screens (WelcomeMock, SpreadsheetMock, NotationMock, ResultatsMock, ExportMock, BaremeEditorMock, CreateProjectModal, PlayerWindow, DetachedNotesWindow). All wrapped by `AppShell` (titlebar + window chrome) + `AppHeader` (interface switcher / tabs).

These mocks are **purely visual** — they take props for state and render different variants. They must carry the same DA tokens as the real desktop app (see below).

**AMV Notation is Windows/Tauri** — `AppShell` renders Win32 controls (Min/Max/Close glyphs on the right of the titlebar), NOT macOS traffic lights. If you see red/yellow/green dots in a diff, that's wrong. The real app's reference frontend lives at `NetsumaInfo/application-bareme-amv-france` — when in doubt about layout/copy, check that repo (e.g. `NotationClipHeader.tsx`, `FloatingPlayerControls.tsx`).

### Landing sections — `src/components/landing/*`

Static marketing sections (`Nav`, `Hero`, `Features`, `TutorialsIntro`, `Shortcuts`, `Download`, `Footer`). May use `motion` (motion.dev) for micro-animations. Not pinned — they scroll normally between tutorial stages.

`Download.tsx` hardcodes the current version + asset URLs (`AMV.Notation_<version>_x64-setup.exe` / `_x64_en-US.msi`). When bumping versions, update `VERSION`, `EXE_URL`, `MSI_URL` constants there + the Hero badge in `Hero.tsx` + `AppShell.tsx` titlebar version string + `Footer.tsx`. There is no auto-fetch from the GitHub releases API.

## Design system

Tokens in `src/styles.css` under `@theme` (Tailwind v4 token block). Color names mirror the desktop app's CSS vars — when re-skinning, change `--color-surface*`, `--color-primary-*`, `--color-app-bg*`, `--color-accent` here and the entire site retunes. Don't introduce ad-hoc colors in components — add a token.

Tailwind v4 is wired via `@tailwindcss/vite` (no `tailwind.config.*` file — config lives in `styles.css`). Utilities like `surface-card`, `glass`, `grain`, `app-shell`, `app-header-trigger` are custom utilities under `@layer utilities`.

Path alias `@/*` → `src/*` (tsconfig + `vite-tsconfig-paths`).

## SSR caveats

- TanStack Start, target `node-server`. `npm run start` requires a prior `npm run build`.
- Anything touching `window`, `document`, `Lenis`, GSAP plugins must be inside `useEffect` or guarded with `typeof window === 'undefined'` — see `TutorialStage.tsx:52`.
- Fonts: Inter loaded via Google Fonts `<link>` in `__root.tsx` head. No font self-hosting.

## Preview / verify

A `.claude/launch.json` is present. Use `preview_start` (Claude Preview MCP) rather than spawning `npm run dev` via Bash — the preview server enables `preview_screenshot`, `preview_snapshot`, etc. for verifying visual changes.
